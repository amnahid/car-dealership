## Overview

This system integrates with ZATCA's Fatoorah e-invoicing platform as required by Saudi Arabia's e-invoicing mandate:

| Phase | Requirement | Status |
|-------|------------|--------|
| Phase 1 | Structured XML, VAT 15%, QR codes, bilingual (AR+EN) | ✅ Implemented |
| Phase 2 | Cryptographic signing, CSID onboarding, real-time clearance/reporting API | ✅ Implemented |

### Invoice Flow

```
Sale Created (Cash / Installment / Rental)
  ↓
processZatcaInvoice()
  ↓
Generate UUID → Build UBL 2.1 XML → SHA-256 Hash
  ↓
Generate QR (TLV encoded, base64)        ← Phase 1
  ↓
Sign XML (ECDSA secp256k1)               ← Phase 2 (if CSID present)
  ↓
Submit to ZATCA API                      ← Phase 2
  ├── Clearance API  (B2B Standard)
  └── Reporting API  (B2C Simplified)
  ↓
Update PIH chain → Save ZATCA fields to sale record
  ↓
Generate PDF (SAR, bilingual, QR embedded)
```

## Phase 1 — Invoice Generation

Phase 1 runs automatically on every invoice, even without Phase 2 credentials.

### What it does

- Generates a ZATCA-compliant UUID for each invoice
- Builds UBL 2.1 XML with all required fields
- Calculates VAT at 15% (Saudi standard rate)
- Encodes QR code (TLV format, 5 tags for Phase 1)
- Embeds QR code in PDF invoice
- Stores all fields in the sale record

### QR Code TLV Tags

| Tag | Field | Phase |
|-----|-------|-------|
| 1 | Seller name | Phase 1 |
| 2 | Seller TRN | Phase 1 |
| 3 | Invoice timestamp (ISO 8601) | Phase 1 |
| 4 | Invoice total (with VAT) | Phase 1 |
| 5 | VAT amount | Phase 1 |
| 6 | XML hash (SHA-256) | Phase 2 |
| 7 | ECDSA signature | Phase 2 |
| 8 | Seller public key | Phase 2 |

### VAT Calculation

```
Base Amount   = Sale Price / 1.15
VAT Amount    = Base Amount × 0.15
Total w/ VAT  = Base Amount + VAT Amount
```

Helper: `calculateVat(totalPrice, vatRate)` in `src/lib/zatca/invoiceService.ts`

## Phase 2 — Onboarding & API Integration

Phase 2 requires obtaining a Cryptographic Stamp Identifier (CSID) from ZATCA.

### Onboarding Steps

#### Step 1: Generate EC Key Pair

Navigate to **Admin → ZATCA Settings → Onboarding** tab and click **Generate Keys**.

This generates an ECDSA secp256k1 key pair and saves the public key to `ZatcaConfig`.

#### Step 2: Generate CSR (Manual Step)

> ⚠️ Node.js cannot generate ZATCA-compliant CSRs with custom OIDs. This step requires `openssl` CLI.

The UI will display an `openssl` command to run. Execute it in your terminal:

```bash
openssl req -new -newkey ec -pkeyopt ec_paramgen_curve:P-256 \
  -keyout private_key.pem -out csr.pem \
  -config zatca_csr.cnf
```

The config file (`zatca_csr.cnf`) will be generated by the onboarding API. It includes ZATCA-required OID extensions:
- `2.16.840.1.114412.1.1` — TRN (Tax Registration Number)
- `2.16.840.1.114412.1.3` — Organization unit / branch
- `2.16.840.1.114412.1.5` — Invoice type codes
- `1.3.6.1.4.1.311.20.2` — Organization identifier
- `2.16.840.1.114412.1.2` — Environment (sandbox/production)

#### Step 3: Request Compliance CSID

POST the CSR to ZATCA sandbox:

```
POST /api/zatca/onboard
{ "action": "request_compliance_csid", "csr": "<base64-pem>" }
```

Stores `csid` and `csidSecret` in `ZatcaConfig`.

#### Step 4: Compliance Check

Run compliance invoices (ZATCA requires 6 standard + 6 simplified test invoices):

```
POST /api/zatca/onboard
{ "action": "compliance_check" }
```

#### Step 5: Request Production CSID

```
POST /api/zatca/onboard
{ "action": "request_production_csid" }
```

Stores `productionCsid` and `productionCsidSecret` in `ZatcaConfig`.

### PIH Chain

Each invoice includes a hash of the previous invoice (`pih` field). This forms an audit chain.

- Initial PIH (ZATCA-specified): `NWZlY2ViNjZmZmM4NmYzOGQ5NTI3ODZjNmQ2OTZjOTljMmYxN2ZiNTVkMzRlYzYzMDMzNjE5YTM0ZGY4YjEwNw==`
- Stored in `ZatcaConfig.pih`, updated after each successful submission

## Invoice Types

| Type | Code | Used For | ZATCA Action |
|------|------|----------|--------------|
| Simplified | 380, subtype 0200000 | B2C (individuals) | Reporting API |
| Standard | 388, subtype 0100000 | B2B (businesses with TRN) | Clearance API |

### Rules

- **Simplified**: QR code mandatory. No buyer TRN required.
- **Standard**: Buyer TRN required (`vatRegistrationNumber` on Customer). Clearance returns signed XML from ZATCA.
- Default: `Simplified` if `invoiceType` not specified in request body.

### Auto-detection

If the customer `customerType === 'Business'` and has a `vatRegistrationNumber`, the sale form pre-selects `Standard`.

## API Reference

### ZATCA Config

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/api/zatca/config` | Get active ZATCA config |
| POST | `/api/zatca/config` | Create new config |
| PUT | `/api/zatca/config` | Update config |

### Onboarding

| Method | Endpoint | Body | Description |
|--------|----------|------|-------------|
| POST | `/api/zatca/onboard` | `{ action: 'generate_keys' }` | Generate EC key pair |
| POST | `/api/zatca/onboard` | `{ action: 'request_compliance_csid', csr }` | Get sandbox CSID |
| POST | `/api/zatca/onboard` | `{ action: 'compliance_check' }` | Run compliance invoices |
| POST | `/api/zatca/onboard` | `{ action: 'request_production_csid' }` | Get production CSID |

### Retry Failed Submissions

| Method | Endpoint | Body | Description |
|--------|----------|------|-------------|
| POST | `/api/zatca/retry` | `{ referenceId, referenceType }` | Retry single invoice |
| POST | `/api/zatca/retry` | `{ batchRetry: true }` | Retry all Failed/Pending invoices |

### Sale APIs (updated)

All sale creation endpoints now accept additional ZATCA fields:

```json
{
  "invoiceType": "Simplified",    // "Standard" | "Simplified"
  "buyerTrn": "123456789012345"   // Required for Standard invoices
}
```

## Troubleshooting

### "No active ZATCA config found"

ZATCA processing will proceed in **Phase 1 only** mode. The invoice gets a UUID and QR code but is not submitted to ZATCA API. Status will be `NotRequired`.

**Fix:** Go to `/dashboard/admin/zatca` and create a config.

### Invoice status stuck at "Pending"

Use the retry endpoint:
```
POST /api/zatca/retry
{ "batchRetry": true }
```

### "CSID not configured" error

Phase 2 signing/submission skipped. Invoice marked as `Pending`.

**Fix:** Complete Phase 2 onboarding steps.

### QR code not showing on PDF

QR code is only embedded if `zatcaQRCode` field is populated. Check:
1. ZATCA config exists (`ZatcaConfig` collection in MongoDB)
2. Sale record has `zatcaQRCode` field set
3. Check server logs for ZATCA processing errors

### XML Schema Validation Failure

Validate against ZATCA XSD schemas. Download from [ZATCA Developer Portal](https://developer.zatca.gov.sa).

Key requirements:
- All monetary amounts must have exactly 2 decimal places
- Timestamps must be ISO 8601 format: `YYYY-MM-DDTHH:MM:SS`
- TRN must be exactly 15 digits

## Security Notes

> ⚠️ **Private Key Storage**  
> Currently, `ZatcaConfig.privateKey` is stored **in plaintext** in MongoDB.  
> Before going to production, encrypt it with AES-256:
> ```
> ZATCA_KEY_SECRET=<random-256-bit-hex>
> ```
> Update `src/models/ZatcaConfig.ts` to use encrypted storage with this secret.

> ⚠️ **Arabic Font in PDF**  
> jsPDF default fonts (Helvetica) do not render Arabic characters correctly.  
> For proper Arabic text, embed an Arabic Unicode font (Amiri or Cairo) in the PDF generator.  
> See `src/lib/invoiceGenerator.ts`.

> ⚠️ **CSID Secrets**  
> `csidSecret` and `productionCsidSecret` are stored in MongoDB.  
> These should also be encrypted or stored in a secrets manager (e.g., AWS Secrets Manager, Azure Key Vault).
